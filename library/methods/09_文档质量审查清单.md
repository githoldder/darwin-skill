# 文档质量审查清单

> 本模块建立文档质量评分模型和标准化审查机制，用于在文档发布前、归档前、复用前进行质量评估。

---

## 一、文档质量评分模型（10分制）

### 评分等级

| 总分 | 等级 | 说明 |
|---|---|---|
| 9.0-10 | 优秀 | 达到正式交付标准，可直接使用 |
| 7.0-8.9 | 良好 | 基本达标，有小问题需修复 |
| 5.0-6.9 | 及格 | 勉强可用，需较大修改 |
| <5.0 | 不及格 | 不可用，需重写 |

### 评分维度权重

| 维度 | 权重 | 说明 |
|---|---|---|
| 结构完整性 | 15% | 章节齐全、必要元素不缺 |
| 内容准确性 | 20% | 事实正确、无技术错误 |
| 逻辑连贯性 | 15% | 章节衔接自然、论述严密 |
| 标准符合性 | 15% | 符合 GB/T 等适用标准 |
| 证据充分性 | 10% | 引用规范、有数据支撑 |
| 可执行性 | 10% | 可直接指导下一步行动 |
| 可维护性 | 5% | 版本清晰、易于更新 |
| 可读性 | 5% | 表达清晰、无歧义 |
| 版本可追踪性 | 5% | 变更记录完整、可回溯 |

**计算公式：**
```
总分 = Σ(维度得分 × 权重) 
```

## 二、各维度评分细则

### 2.1 结构完整性（15%）

| 分值 | 说明 |
|---|---|
| 13-15 | 所有必要章节齐全，有摘要、目录、参考文献、变更记录 |
| 10-12 | 必要章节齐全，缺少1-2个非核心元素 |
| 6-9 | 缺少多个重要章节（如无摘要、无参考文献） |
| <6 | 结构严重缺失，无法构成完整文档 |

**检查项：**
- [ ] 有标题（标题准确反映内容）
- [ ] 有摘要（学术/论文类文档）
- [ ] 有目录
- [ ] 有章节结构
- [ ] 有参考文献（引用类文档）
- [ ] 有变更记录（正式文档）
- [ ] 有作者/日期/版本信息

### 2.2 内容准确性（20%）

| 分值 | 说明 |
|---|---|
| 18-20 | 所有事实正确，技术细节准确，无错误 |
| 14-17 | 事实基本正确，有少量不影响理解的小错误 |
| 10-13 | 有多处事实错误，需要核对 |
| <10 | 内容错误较多，需要全面重写 |

**检查项：**
- [ ] 技术描述准确（算法、接口、数据结构等）
- [ ] 数据、统计数字准确
- [ ] 引用标准正确（标准号、名称、实施日期）
- [ ] 无拼写错误
- [ ] 无混淆概念
- [ ] 代码/命令可运行/可执行

### 2.3 逻辑连贯性（15%）

| 分值 | 说明 |
|---|---|
| 13-15 | 论述逻辑严密，章节衔接自然，无跳跃 |
| 10-12 | 整体逻辑清晰，有1-2处轻微跳跃 |
| 6-9 | 有明显逻辑断层，需要补充过渡 |
| <6 | 逻辑混乱，章节之间缺乏关联 |

**检查项：**
- [ ] 绪论/背景与正文内容对应
- [ ] 每章有承上启下的过渡
- [ ] 因果关系论述清晰
- [ ] 无自相矛盾的论述
- [ ] 对比/分析类文档结构合理

### 2.4 标准符合性（15%）

| 分值 | 说明 |
|---|---|
| 13-15 | 符合所有适用的 GB/T 或 ISO/IEC/IEEE 标准 |
| 10-12 | 符合主要标准，有1-2处小偏差 |
| 6-9 | 标准符合性较差，偏离了多个标准要求 |
| <6 | 完全不符合标准要求 |

**检查项：**
- [ ] 文档类型匹配了正确的标准
- [ ] 格式符合标准（字体、字号、行距）
- [ ] 图表编号符合标准
- [ ] 引用格式符合 GB/T 7714—2015
- [ ] 文档结构符合对应标准章节要求
- [ ] 量和单位使用符合 GB/T 3100 系列

### 2.5 证据充分性（10%）

| 分值 | 说明 |
|---|---|
| 9-10 | 所有重要结论都有引用支撑，引用质量高 |
| 7-8 | 大部分结论有引用，有少量未经论证的结论 |
| 5-6 | 引用不足，大量结论缺少支撑 |
| <5 | 几乎无引用，不可作为正式文档 |

**检查项：**
- [ ] 每个重要结论都有引用
- [ ] 引用来源可信（L1-L3）
- [ ] 引用格式正确
- [ ] 正文中标注了引用编号
- [ ] 无 AI 生成内容作为事实来源

### 2.6 可执行性（10%）

| 分值 | 说明 |
|---|---|
| 9-10 | 可直接指导开发/测试/答辩/验收等行动 |
| 7-8 | 可指导大部分行动，有少量模糊之处 |
| 5-6 | 需补充才能指导行动 |
| <5 | 无法指导行动，仅供参考 |

**检查项：**
- [ ] 需求规格包含验收标准
- [ ] 操作步骤具体可执行
- [ ] 参数、配置说明具体
- [ ] 有边界条件和异常处理说明
- [ ] 测试用例可执行（有输入输出）

### 2.7 可维护性（5%）

| 分值 | 说明 |
|---|---|
| 5 | 命名规范、版本清晰、易于更新 |
| 4 | 基本满足可维护性，有小问题 |
| 3 | 命名或版本混乱，维护困难 |
| <3 | 基本不可维护 |

### 2.8 可读性（5%）

| 分值 | 说明 |
|---|---|
| 5 | 语言表达精准，无歧义，易读 |
| 4 | 基本可读，有少量歧义表达 |
| 3 | 有多处歧义或表述不清 |
| <3 | 表达混乱，难以理解 |

### 2.9 版本可追踪性（5%）

| 分值 | 说明 |
|---|---|
| 5 | 变更记录完整，可追踪每次变更 |
| 4 | 有基本变更记录 |
| 3 | 变更记录不完整 |
| <3 | 无变更记录 |

## 三、一票否决项

以下问题出现**任意一项**，直接判定为**不合格**，不进入评分：

| 序号 | 否决项 | 说明 |
|---|---|---|
| 1 | **抄袭/大段复制未标注** | 违反学术/版权规范 |
| 2 | **引用虚假/不存在的标准** | 伪造信息来源 |
| 3 | **核心章节内容严重缺失** | 如毕设第3-4章凑字数 |
| 4 | **技术方案包含明显错误** | 会导致开发/测试误导 |
| 5 | **参考文献格式完全不统一** | 超过50%格式错误 |
| 6 | **未完成文档冒充完成** | 如SRS未覆盖核心需求 |
| 7 | **严重违反对应国家标准章节要求** | 结构上不符合标准 |
| 8 | **AI 生成内容作为事实来源** | 违反引用规范 |

## 四、常见问题库（20个）

| 序号 | 问题 | 原因 | 解决方案 |
|---|---|---|---|
| 1 | 摘要与正文内容不一致 | 摘要和正文分开写未核对 | 写完正文后重新核对摘要 |
| 2 | 图表编号不连续 | 中途增删图表 | 全文搜索验证编号 |
| 3 | 正文中引用了图表但图不存在 | 删图忘删引用 | 全文搜索验证引用 |
| 4 | 参考文献格式不统一 | 使用了多种文献格式 | 用 Zotero/EndNote 重新格式化 |
| 5 | 章节标题过于笼统 | 写作时未精炼标题 | 改为描述本章具体内容 |
| 6 | 关键算法描述模糊 | 怕写错刻意模糊 | 补充伪代码或具体步骤 |
| 7 | 需求规格无法测试 | 需求无法量化 | 改写为可验证的描述 |
| 8 | 数据库字段与需求不一致 | 设计变更未同步 | 核对需求文档和数据库设计 |
| 9 | 接口参数与实际代码不符 | 代码改了文档没改 | 建立文档与代码同步机制 |
| 10 | 测试用例覆盖不足 | 赶进度跳过测试用例 | 补充边界条件和异常用例 |
| 11 | 文档命名随意 | 无规范意识 | 建立命名规范并执行 |
| 12 | 版本号混乱 | 未按语义化版本管理 | 建立版本号管理制度 |
| 13 | 文档之间相互矛盾 | 不同人写不同文档未对齐 | 建立跨文档评审机制 |
| 14 | 过度引用AI生成内容 | 依赖AI而未核实 | 建立AI使用声明和复核机制 |
| 15 | 文档过期未更新 | 写完后不再维护 | 建立文档有效期审查机制 |
| 16 | 流程图与实际流程不符 | 流程变了图没改 | 流程变更时同步更新图 |
| 17 | 缺少变更记录 | 多次修改未记录 | 每次修改更新CHANGELOG |
| 18 | 文档无读者定位 | 写给谁看不明确 | 明确每份文档的目标读者 |
| 19 | 表格跨页不断行 | 格式不规范 | 设置表格跨页断行规则 |
| 20 | 术语使用不统一 | 同义词混用 | 建立术语表并保持一致 |

## 五、AI 辅助审查 Prompt

将以下 Prompt 用于对文档进行 AI 辅助质量审查：

```
你是一名专业的文档质量审查员。请对以下文档进行质量审查。

审查标准：
1. 结构完整性：检查是否包含所有必要章节（标题、摘要、目录、正文、参考文献、变更记录）
2. 内容准确性：检查技术描述是否准确，数据是否可信
3. 逻辑连贯性：检查章节衔接是否自然，论述是否有逻辑断层
4. 标准符合性：检查是否符合 GB/T 7714—2015（引用格式）、GB/T 7713.1—2025（学位论文）或 GB/T 9385—2008（需求规格）
5. 证据充分性：检查每个重要结论是否有引用支撑
6. 可执行性：检查是否可直接指导下一步行动
7. 可维护性：检查命名、版本、变更记录是否规范
8. 可读性：检查语言是否精准、表达是否清晰

请按以下格式输出审查结果：

【质量评分】（10分制）
【各维度得分】（结构15 + 准确20 + 逻辑15 + 标准15 + 证据10 + 执行10 + 维护5 + 可读5 + 追踪5）
【一票否决项检查】（列出是否有否决项）
【主要问题】（列出最重要的3-5个问题）
【改进建议】（针对每个主要问题给出具体改进建议）
【总体评价】（总结性评价，是否适合发布/归档/复用）

---
以下是需要审查的文档内容：
[paste document here]
```

## 七、文档发布前检查清单

在文档正式发布前，必须完成以下所有检查项：

### A. 格式检查
- [ ] 字体、字号、行距符合模板/标准要求
- [ ] 页眉页脚格式正确
- [ ] 页码正确（封面、目录不编页码，正文从第1页开始）
- [ ] 图表编号连续，无重复无跳跃
- [ ] 所有图表有题注
- [ ] 公式有编号
- [ ] 代码块格式正确

### B. 内容检查
- [ ] 正文内容与标题一致
- [ ] 各章节内容饱满（无凑字数章节）
- [ ] 重要结论有数据/引用支撑
- [ ] 无自相矛盾的内容
- [ ] 无抄袭（大段复制须标注）

### C. 引用检查
- [ ] 所有引用在正文中标注了编号
- [ ] 所有正文引用在参考文献中有对应条目
- [ ] 参考文献格式符合 GB/T 7714—2015
- [ ] 引用来源可信（L1-L3）
- [ ] 无 AI 生成内容作为事实来源

### D. 标准检查
- [ ] 文档类型匹配了正确的标准
- [ ] 结构符合标准章节要求
- [ ] 非功能需求可量化
- [ ] 术语使用统一

### E. 元数据检查
- [ ] 有完整的元数据头
- [ ] 版本号正确
- [ ] 作者/审核人已填写
- [ ] 日期正确

### F. 可执行性检查（仅限需求/设计文档）
- [ ] 需求规格可测试
- [ ] 设计文档可指导编码
- [ ] 测试用例可执行

## 八、文档归档前检查清单

- [ ] 文档状态为 `approved`
- [ ] 所有审查意见已处理
- [ ] 变更记录完整
- [ ] 归档路径符合目录规范
- [ ] 元数据完整（存储路径已填）
- [ ] 关联文档已建立链接
- [ ] 归档命名符合规范
- [ ] 已上传/存入指定位置
- [ ] 索引记录已创建

---

## 九、文档复用前检查清单

- [ ] 确认文档适用范围包含当前项目
- [ ] 确认文档版本与当前需求兼容
- [ ] 文档状态为 `approved` 或 `archived`（非 draft/deprecated）
- [ ] 检查文档是否过期（超过有效期需重新审核）
- [ ] 引用了已废弃的旧版本 → 更新引用
- [ ] 记录复用来源（便于追溯）

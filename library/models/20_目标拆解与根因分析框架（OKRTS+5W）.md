# 目标拆解与根因分析框架（OKRTS + 5W + 问题四象限）

---

## 框架总览

本框架整合了三种核心思维模型，覆盖从**目标制定 → 任务拆解 → 问题诊断 → 分类应对**的完整闭环：

| 模型 | 核心功能 | 适用场景 |
|------|----------|----------|
| **OKRTS 金字塔** | 大目标逐级拆解到可执行步骤 | 目标制定、项目规划、任务分解 |
| **5W 根因分析法（漏斗模型）** | 从表层现象深挖根因 | 问题诊断、复盘分析、故障排查 |
| **问题关注四象限** | 按复杂度/深度分类问题，匹配应对策略 | 优先级排序、资源分配、决策制定 |

---

## 模型一：OKRTS 金字塔模型（目标逐级拆解）

### 模型结构

```
                    ┌─────────────┐
                    │   O (Object) │  ← 大目标（愿景、价值）
                    │   目标/愿景   │
                    └──────┬──────┘
                           │  ↓ 逐级拆解
                    ┌──────▼──────┐
                    │ KR (Key     │  ← 关键结果（里程碑、关键路径）
                    │  Result)    │
                    │  关键阶段    │
                    └──────┬──────┘
                           │  ↓
                    ┌──────▼──────┐
                    │  T (Task)   │  ← 具体任务（细到具体的可行任务）
                    │  具体任务    │
                    └──────┬──────┘
                           │  ↓
                    ┌──────▼──────┐
                    │  S (Step)   │  ← 详细步骤（每个2分钟内可被执行）
                    │  详细步骤    │
                    └─────────────┘
```

### 层级说明

| 层级 | 名称 | 核心问题 | 特征 | 示例 |
|------|------|----------|------|------|
| **O** | Object（目标） | 我们为什么要做？ | 愿景驱动、价值导向、较模糊 | "打造行业领先的开发者工具" |
| **KR** | Key Result（关键结果） | 如何衡量达成？ | 里程碑式、可量化、关键路径 | "月活用户突破10万" |
| **T** | Task（任务） | 具体做什么？ | 可执行、有明确产出、细粒度 | "完成用户注册功能的开发" |
| **S** | Step（步骤） | 当前做什么？ | **2分钟内可执行**、原子化操作 | "编写注册表单的HTML结构" |

### 执行原则

1. **自上而下拆解**：从 O 开始逐级细化，确保每一级都与上一级对齐
2. **S 步骤原子化**：最底层的 Step 必须保证任何人在 2 分钟内可执行完毕
3. **认知资源最大化利用**：通过逐级拆解，将模糊的大目标转化为清晰的行动路径
4. **模糊 → 清晰**：拆解的过程就是将不确定性逐步消除的过程

### 操作清单

- [ ] **设定 O**：明确大目标，回答"为什么做"
- [ ] **定义 KR**：拆解 2-4 个关键结果作为里程碑
- [ ] **规划 T**：为每个 KR 列出具体的执行任务
- [ ] **细化 S**：将任务进一步拆解为原子化步骤（2分钟法则）
- [ ] **对齐检查**：确认每一级拆解都与上层目标一致

## 模型二：5W 根因分析法（漏斗模型）

### 模型结构

```
                   ┌─────────────────┐
                   │    表面现象      │  ← 观察到的直接问题
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │  Why1: 直接原因   │  ← 最表层的原因
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │  Why2: 前置条件   │  ← 导致直接原因的诱因
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │  Why3: 执行偏差   │  ← 执行层面的问题
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │  Why4: 认知漏洞   │  ← 认知/知识层面的问题
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │  Why5: 根问题     │  ← 最深层、最根本的原因
                   └────────┬────────┘
                            │ ↓
                   ┌────────▼────────┐
                   │    解决方案      │  ← 基于根因制定对策
                   └─────────────────┘
```

### 逐层追问指南

| 层级 | 追问方向 | 典型问题 |
|------|----------|----------|
| **表面现象** | 发生了什么？ | "具体观察到什么问题？" |
| **Why1: 直接原因** | 最直接的技术/操作原因？ | "为什么这个现象会发生？" |
| **Why2: 前置条件** | 什么样的环境/条件导致了它？ | "这个原因是在什么条件下产生的？" |
| **Why3: 执行偏差** | 流程/执行层面出了什么问题？ | "为什么流程没有拦住这个问题？" |
| **Why4: 认知漏洞** | 知识/认知层面缺了什么？ | "团队成员缺少什么必要的认知？" |
| **Why5: 根问题** | 更深层的系统/文化/机制问题？ | "整个系统/组织中根本的问题是什么？" |

### 执行要点

- **追问到底**：连续追问 5 层 Why（实际可能需要更多或更少），直到找到可行动的根因
- **表层 → 深层**：每深入一层，问题范围更广、影响更深远、解决方案也更具系统性
- **方案匹配根因**：解决方案必须直接针对根因（Why5），而非表层现象
- **记录分析路径**：保留完整的 5W 分析链条，便于后续复盘和知识沉淀

## 模型三：问题关注四象限模型（分类应对）

### 四象限结构

```
                     深 层（Deep）
                         ↑
               ┌─────────┼─────────┐
               │         │         │
               │  P2     │  P1     │
               │ 简单深层│ 复杂深层│
               │ 问题    │ 问题    │
               │         │         │
   简单 ───────┼─────────┼─────────┼──────→ 复杂
   (Simple)    │         │         │    (Complex)
               │  P4     │  P3     │
               │ 简单浅层│ 复杂浅层│
               │ 问题    │ 问题    │
               │         │         │
               └─────────┼─────────┘
                         │
                         ↓
                    浅 层（Shallow）
```

### 问题分类与应对策略

| 象限             | 特征                    | 应对策略                                        | 实际占比    |
| -------------- | --------------------- | ------------------------------------------- | ------- |
| **P1: 复杂根问题**  | 需要深入分析、依赖经验判断、方案设计难度大 | **慎重决策、冷静判断**<br>调研对比多种解决方案<br>依赖资深工程师的专业能力 | **5%**  |
| **P2: 简单深层问题** | 根因较深但逻辑清晰、有标准解法       | **脚本自动化、代理/外包处理**<br>制定标准化流程<br>一次性投入解决     | **15%** |
| **P3: 复杂浅层问题** | 涉及面广但不需要深挖、有大量细节      | **文档化、录制操作视频**<br>建立知识库/FAQ<br>通过培训降维解决     | **30%** |
| **P4: 简单浅层问题** | 一眼可见、操作简单、重复性高        | 快速处理或直接忽略<br>（低价值、低复杂度）                     | **50%** |

### 资源分配原则

```
资源投入比例（精力/时间/预算）：

P1（复杂根问题） → 50%
P3（复杂浅层问题） → 30%
P2（简单深层问题） → 15%
P4（简单浅层问题） → 5%
```

- **优先投入 P1**：最有价值但也最困难，需要深度思考+专业判断
- **系统化解决 P3**：通过文档化、标准化降低复杂度
- **自动化处理 P2**：一次性投入换长期收益
- **快速过滤 P4**：不消耗过多认知资源

### 使用流程

1. **定位问题**：判断问题在四象限中的位置（简单/复杂 × 浅层/深层）
2. **匹配策略**：根据象限选择对应的应对方式
3. **分配资源**：按比例分配精力，避免在低价值问题上过度投入
4. **动态调整**：随着认知加深，问题可能跨象限移动，及时调整策略

## 三模型联动工作流

在实际工作中，三个模型可以组合使用，形成完整闭环：

```
         ┌──────────────────────────────────────────┐
         │           Step 1: 目标设定               │
         │    使用 OKRTS 金字塔设定 O 并拆解 KR      │
         └──────────────────┬───────────────────────┘
                            │
         ┌──────────────────▼───────────────────────┐
         │           Step 2: 执行与监控              │
         │    使用 OKRTS 继续拆解 T → S 并执行       │
         └──────────────────┬───────────────────────┘
                            │
              遇到问题 ←────┘
                │
         ┌──────▼───────────────────────────────────┐
         │           Step 3: 问题诊断               │
         │  使用 5W 漏斗模型从表层挖到根因           │
         └──────────────────┬───────────────────────┘
                            │
         ┌──────────────────▼───────────────────────┐
         │           Step 4: 分类应对               │
         │  使用问题四象限定位问题类型，匹配策略       │
         └──────────────────┬───────────────────────┘
                            │
         ┌──────────────────▼───────────────────────┐
         │           Step 5: 调整目标/任务           │
         │    回到 Step 1，更新 OKRTS 拆解           │
         └──────────────────────────────────────────┘
```

### 典型场景示例

**场景：项目延期**

1. **OKRTS 检查**：核对当前执行的 T/S 是否完整对齐 KR/O
2. **5W 根因分析**：
   - Why1：某个功能模块未按时完成（直接原因）
   - Why2：前期需求评估不足（前置条件）
   - Why3：未进行阶段评审（执行偏差）
   - Why4：团队低估了技术复杂度（认知漏洞）
   - Why5：缺乏技术预研和风险评估机制（根问题）
3. **四象限定位**：这是一个 **P1（复杂根问题）** → 需要系统性的流程改进
4. **策略执行**：投入 50% 精力建立技术预研和风险评估机制
5. **OKRTS 调整**：将"建立技术评估流程"作为新的 KR，拆解到可执行的 T/S

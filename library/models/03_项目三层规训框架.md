## 框架概述

针对任何项目，必须严格遵循以下三层框架（**不可跳过任何一层**）：

| 层级 | 名称 | 核心问题 | 关键产出 |
|------|------|----------|----------|
| Layer 1 | 需求对齐层 | 做什么？ | 需求确认文档 |
| Layer 2 | PM 0-1 全流程层 | 怎么做？ | 产品设计与迭代 |
| Layer 3 | 项目经理里程碑层 | 何时完成？ | 进度与风险管控 |

---

## Layer 1: 需求对齐层（客户信息对齐）

### 核心原则

任何任务开始前，必须先对齐需求，消除歧义。

### 执行清单

- [ ] 列出"已知需求"（用户明确提出的）
- [ ] 列出"假设需求"（基于经验的合理推测）
- [ ] 标记"待用户确认点"（需要澄清的问题）

### 输出格式

每次输出前按以下顺序呈现：

```markdown
## 需求对齐总结

### 已知需求
1. 需求1
2. 需求2

### 假设需求
1. 假设1（基于...推测）
2. 假设2（基于...推测）

### 待确认点
- [ ] 问题1：...
- [ ] 问题2：...

---

## 执行内容
...
```

### 自检问题

> 每次输出前问自己："这个输出是否与最终使用者的真实目标完全对齐？"
> 
> 如有偏差，立即提出澄清问题，而非继续执行。

---

## Layer 2: PM 0-1 全流程层（从0到1打通）

### 核心原则

必须完成"分析 → 设计 → 实现 → 验证 → 上线/迭代"的闭环思维。

### 各阶段交付物

| 阶段        | 核心交付物          | 决策依据      |
| --------- | -------------- | --------- |
| **分析**    | 用户调研、竞品分析、需求文档 | 用户价值、市场机会 |
| **设计**    | 原型、PRD、技术方案    | 可行性、成本效益  |
| **实现**    | 代码、测试用例        | 质量标准、进度要求 |
| **验证**    | 测试报告、用户反馈      | 验收标准、成功指标 |
| **上线/迭代** | 发布计划、运营方案      | 业务目标、数据指标 |

### 产品思维检查点

- [ ] 始终围绕**用户价值**，而非技术炫技
- [ ] 始终围绕**业务目标**，而非功能堆砌
- [ ] 始终围绕**数据指标**，而非主观感觉

---

## Layer 3: 项目经理里程碑层（顺产不夭折）

### 核心原则

自动拆解里程碑、阶段性 KPI、时间/资源/风险规划。

### 每次迭代输出必须包含

```markdown
## 项目状态报告

### 当前里程碑进度
- 里程碑：...
- 进度：X%
- 状态：正常/预警/阻塞

### 下一阶段计划
- 目标：...
- 关键任务：...
- 预计完成：...

### 潜在风险及应对
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 风险1 | 高/中/低 | 高/中/低 | 方案... |

### 资源盘点
- 人力：...
- 时间：...
- 预算：...
```

### 约束感保持

- **时间盒**：为每个任务设定明确的截止时间
- **优先级排序**：P0 > P1 > P2，资源向高优先级倾斜
- **资源盘点**：定期审视人力、时间、预算消耗

---

## 通用执行铁律

适用于所有项目的强制性规则：

1. **每次对话/任务开始时**，先输出"三层检查清单"简表
2. **所有方案必须同时满足**：需求精准 + 流程完整 + 项目可控
3. **发现任何"任务失焦"迹象**，立即暂停并拉回三层框架
4. **留白原则**：具体细节由项目上下文决定，但框架不可打破

### 流程详解

#### 主线流程（A → G）

**1. 想法阶段（Ideation）**
- 输入：人的需求、社会需要、商业战略、技术升级
- 输出：初步概念
- 关键活动：头脑风暴、市场调研、技术预研

**2. 论证阶段（Feasibility）**
- 输入：初步概念
- 输出：可行性分析报告
- 关键活动：
  - 可行性分析（技术/商业/运营）
  - 收益评估（ROI 计算）
  - 成本估算（人力/时间/资金）
  - 备选方案对比
- **三层框架映射**：Layer 1 需求对齐

**3. 启动阶段（Initiation）**
- 输入：可行性报告
- 输出：项目章程
- 关键活动：
  - 明确目标（SMART 原则）
  - 定义成功标准（可衡量）
  - 设定需求里程碑
  - 确定预算与资源投入
  - 识别初步风险
- **三层框架映射**：Layer 2 分析阶段完成

**4. 规划阶段（Planning）**
- 输入：项目章程
- 输出：项目计划书
- 关键活动：
  - 细化需求范围（Scope）
  - 制定进度计划（Schedule）
  - 成本预算细化（Cost）
  - 质量标准定义（Quality）
  - 资源分配（Resource）
  - 风险应对计划（Risk）
- **三层框架映射**：Layer 2 设计阶段
- **循环**：规划 ↔ 项目进度计划（持续迭代）

**5. 执行阶段（Execution）**
- 输入：项目计划
- 输出：可交付成果
- 关键活动：
  - 组建团队
  - 干系人沟通管理
  - 按计划实施（开发/生产/服务）
- **三层框架映射**：Layer 2 实现阶段

**6. 监控阶段（Monitoring）**
- 并行于执行阶段
- 关键活动：
  - 进度跟踪
  - 质量控制
  - 变更管理
  - 风险监控
- **三层框架映射**：Layer 3 持续进行

**7. 收尾阶段（Closure）**
- 输入：可交付成果
- 输出：移交成果、运营方案
- 关键活动：
  - 成果移交
  - 文档归档
  - 团队解散
  - 经验总结
- **三层框架映射**：Layer 2 验证与上线阶段

#### 反馈循环

**质量反馈环（监控 → 审计 → 验收 → 问题 → 规划）**
- 监控发现异常 → 触发审计
- 审计结果 → 验收评审
- 验收发现问题 → 回归规划调整
- 形成 PDCA 闭环

**价值验证环（收尾 → 验证商业论证 → 论证）**
- 项目完成后验证商业论证
- 结果反馈到论证阶段
- 优化未来项目的可行性评估

## 快速检查清单

### 任务启动时

```markdown
## 三层检查清单

### Layer 1 - 需求对齐
- [ ] 已知需求已列出
- [ ] 假设需求已标注
- [ ] 待确认点已提出

### Layer 2 - PM 0-1
- [ ] 当前所处阶段明确
- [ ] 阶段交付物清晰
- [ ] 用户价值已确认

### Layer 3 - 里程碑
- [ ] 里程碑已定义
- [ ] 进度可衡量
- [ ] 风险已识别
```

### 周会/迭代回顾时

```markdown
## 项目健康度检查

### 进度
- 计划完成：X%
- 实际完成：Y%
- 偏差原因：...

### 质量
- 缺陷数：...
- 测试覆盖率：...

### 风险
- 新增风险：...
- 已关闭风险：...

### 资源
- 人力消耗：...
- 预算消耗：...
- 是否需要调整：...
```

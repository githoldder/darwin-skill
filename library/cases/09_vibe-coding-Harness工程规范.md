> 来源：harness_engineering（Ralph组件）.excalidraw

## AI Agent 长时间运行的四大问题

| 问题 | 说明 |
|------|------|
| **上下文窗口有限** | 上下文越长 → 注意力分散 → 指令遵循越弱 → 越傻 |
| **没有持久记忆** | 重启 session 全忘，压缩上下文也不靠谱（不知道什么信息对解决问题重要） |
| **错误滚雪球** | AI 改错地方如果 AC，就会导致一开始导致问题出现的"根问题"一直在发力，从而导致一步错步步错 |
| **AI 天生爱偷懒** | 写个 `//TODO: implement later` 糊弄你 |

### 根本原因

> 上下文有限 **vs** 真实项目需要持续大量信息处理

## Ralph 组件核心原理

**一个 bash 循环**，重复最多 25 次：

```
把指令文件 prompt.md 发送给 AI Agent
    ↓
等它干完活
    ↓
如果 AI 回复"全部完成" → 停下来
否则 → 下一轮
```

### 每轮指令

1. 读任务清单（`prd.json`），找下一个没完成的任务
2. 读经验日志（`progress.txt`），看看前几轮踩了什么坑
3. 实现这一个任务
4. 跑测试
5. 测试通过 → 提交代码，标记任务完成
6. 把这轮学到的东西写进经验日志
7. 如果所有任务都完成了 → 回复"全部完成"

### 核心优势

> **每轮全新上下文，用完就扔，每轮都是最清醒的状态**

## 三件套记忆

| 组件 | 作用 |
|------|------|
| **任务清单** | 做到哪了 |
| **经验日志** | 踩过什么坑 |
| **Git 提交** | 每轮代码成果 |

## 实操环节

### Step 1 — 准备阶段

1. 提示其到 GitHub 上下载 Ralph 项目
2. 一句话告诉他用户的目的，并且生成 PRD 和 MVP 方案

### Step 2 — 任务拆分

3. `prd.md` 以"User Story"为线索和地图，将每一个细节任务拆分成能被高质量处理的"原子化任务"
4. 将 `prd.md` 转化成 `prd.json`；Ralph 项目自带拆分功能，将 Story 拆分成单次迭代可完成的小任务

### Step 3 — 执行迭代

5. 使用 Git 本地建仓并创建分支，然后逐一执行

---

## 重点注意事项

> [!IMPORTANT]
> **任务拆分得越小越好**，一轮干不完就开始糊弄

> [!IMPORTANT]
> **验收标准写得越死越好**，越模糊 AI 钻空子余地越大

> [!IMPORTANT]
> **Git 是安全网**，遇到 bug 就 reset，最多丢一轮，不会错误累计

## 工作流图示

```
┌─────────────────────────────────────────────┐
│              Ralph Bash Loop                │
│               (最多 25 次)                  │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│ 1. 读任务清单 prd.json → 找下一个未完成任务  │
│ 2. 读经验日志 progress.txt → 避坑           │
│ 3. 实现这一个任务                            │
│ 4. 跑测试                                    │
│ 5. 测试通过 → Git 提交 → 标记完成            │
│ 6. 写经验日志 progress.txt                   │
│ 7. 所有任务完成？→ 回复"全部完成"            │
└──────────┬──────────────────────────────────┘
           │
           ▼
    ┌──────────────┐
    │ "全部完成" ?  │
    └────┬─────────┘
      Yes → 退出循环
      No  → 下一轮（全新上下文）
```
